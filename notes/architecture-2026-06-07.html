Current architecture — voice agent + background supervisor (post-milestone B)

As of 2026-06-07. SOP-constrained voice dialogue with pool-based speculative cache, cosine-only rerank, and instruction prefetch.

What's on the critical path (per turn)

  • Step 1 — Classifier + proposer: one fast-model LLM call. Inputs: live user message, full history, SOP-allowed actions, cohort/mood vocabularies, retrieved precedents. Outputs: classified cohort/state/mood, top-K candidate next-actions. ~1.5 s.
  • Step 2 — Pool rerank: no LLM. Computes embedding of live message, cosines against pre-stored embeddings of pool's data items, dedups, returns top-3 to attach to step 4's prompt. ~300 ms mean, 483 ms p95.
  • Step 3 — Instruction lookup (milestone B): no LLM. Exact-match on (chosen_action, classified_state) against pool's instruction items. Hit → use payload verbatim and skip step 4. < 1 ms.
  • Step 4 — response_gen: one fast-model LLM call. Pool-data picks attach as PREFETCHED CONTEXT block with speculative-context framing. ~1.5 s. Skipped entirely on instruction hit.

What runs in the background (async lane)

  • Triggered at every live turn-end. The live agent never blocks on this lane.
  • Empirical predictor (workhorse): SQL over precedent_traces for (cohort, state) → action distribution at offsets 1–3. Cheap, deterministic, scales with deployment data. Drives 88% of pool fetches.
  • MCTS pondering: depth-3 rollouts with per-cohort mood diversity. Imagines future trajectories. Architecturally supported and verified to route correctly; in autopilot conditions 98% of runs are cancelled by the next turn before they finish, so its contribution is 3%. Real voice calls (2–5 s pauses) should give it more room.
  • Data prefetch fan-out: schedules speculative fetches against external data sources (Policy DB, Claims RAG, Market KG, Discount API) keyed by the predicted (action, dep_name) tuples. Results land in the pool as kind="data" items.
  • Instruction prefetch (POC): pre-runs response_gen for the empirically-likely (next-action, next-state) and stores the produced text as kind="instruction" pool item. Consumed via exact-match by step 3. Fix A (2026-06-07): before generating, it pulls the pool's data items relevant to the predicted action (its declared dependencies, or items fetched under it) and attaches them — so the pre-generated instruction is data-informed. Without this, an instruction hit at the live turn would skip response_gen and silently discard the prefetched data.

Fix A — closing the data ↔ instruction gap

  • The gap: on an instruction hit, step 4 (response_gen) is skipped — and with it, the data picks that step 2 would have injected. With a 60% hit rate, that's 60% of turns where the data prefetch work was orphaned.
  • The fix: the instruction-generation step (background) now reads the pool's relevant data items and bakes them into the pre-generated text. On hit, the agent's verbatim response already reflects the prefetched POLICY / CLAIMS / MARKET / DISCOUNTS data.
  • Net: the data path and the instruction path are now coherent. The data prefetch is useful on both instruction-hit (via the pre-baked instruction) and instruction-miss (via step 4's prompt) turns.

What data is available on each turn

  • From the user: the new utterance (live message).
  • From history: every prior turn with action tag.
  • From the SOP graph: allowed next-actions, data dependencies per action, cohort × mood vocabularies, user-state vocabulary.
  • From precedent traces: cross-session distributions over (cohort, state, action) used by the empirical predictor for both data and instruction prefetch.
  • From the pool: items deposited by the supervisor on prior turns. Data items participate in step 2 rerank; instruction items participate in step 3 exact-match.

Current measured latency (no-tier-3 + G + v3 + milestone B, autopilot conditions)

  • Pool rerank (step 2): 300 ms mean, 483 ms p95 across N=10. Hit rate 100%.
  • Instruction lookup (step 3): < 1 ms. Hit rate 60% on the smoke session (POC, not yet a clean measurement).
  • response_gen (step 4): ~1.5 s when it fires. Skipped on instruction hit, saving the full call cost.
  • Per-turn agent total: ~4–6 s baseline, varies with how much MCTS/empirical work fires after each turn (next-turn prefetch).
  • p95 target: was 800 ms for rerank, met at 483 ms.

How this differs from the original blog architecture diagram

  • Live MCTS is no longer on the critical path — moved to background pondering only.
  • Pool rerank dropped its LLM call entirely (cosine + dedup only).
  • Empirical predictor is now identified as the workhorse (was framed as one of several equal predictors).
  • Instruction prefetch is new — adds kind="instruction" pool items and a step 3 lookup that can skip step 4 entirely.
  • Latency annotations match the current N=10 measurement.
